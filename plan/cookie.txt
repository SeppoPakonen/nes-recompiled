### FIX APPLIED: MMC1 Fixed Bank Region (2026-03-13)

**Root Cause:**
The recompiler did not handle MMC1's fixed bank region. For MMC1:
- $8000-$BFFF: Switchable PRG ROM (banks 0-14)
- $C000-$FFFF: Fixed to LAST bank (bank 15)

**File Modified:**
`/common/active/sblo/Dev/nes-recompiled/recompiler/src/rom.cpp` - `read_banked()` function

**Fix Applied:**
Added mapper-aware address translation:
```cpp
// For MMC1 (mapper 1):
// $8000-$BFFF: Switchable PRG ROM
// $C000-$FFFF: Fixed to LAST bank
if (mapper == 1 && addr >= 0xC000) {
    // Fixed bank region - always use last bank
    size_t last_bank = header_.rom_banks - 1;
    size_t offset = 16 + (last_bank * 0x4000) + (addr - 0xC000);
    if (offset < data_.size()) {
        return data_[offset];
    }
    return 0xFF;
}
```

**Results After Fix:**
- NMI handler (0xCFF0) now reads correct code: PHA, PHP, TXA... (not 0xFF garbage)
- Reset handler (0xFFE0) now reads correct code: JMP $F2D1 (not BRK)
- Functions in $C000-$FFFF region now properly analyzed
- Generated C code includes func_cff0, func_ffe0, func_c000, etc.

**Before Fix:**
- Total functions: 2 (only vectors, reading garbage)
- Instructions: 11 (garbage data)
- Runtime: Crashed to 0x0000 (BRK vector)

**After Fix:**
- Total functions: 9 (real code from fixed bank)

### Workflow

The trace-guided analysis is most useful for:
1. ROMs where static analysis misses code due to indirect jumps
2. ROMs with runtime-loaded code (e.g., level data with embedded scripts)
3. ROMs with complex bank-switching behavior
4. ROMs where entry points are determined by game state

For simple ROMs like a.nes (16KB NROM), static analysis already finds all reachable code.

## Investigation: Why Only 7 Functions in 128KB ROM (2026-03-12)

### Root Cause Found: NES Memory Map Misunderstanding

The recompiler was designed for GameBoy memory map but used for NES without adapting the memory map logic.

**GameBoy Memory Map:**
- `$0000-$7FFF`: ROM (with banking)
- `$8000-$FFFF`: VRAM, WRAM, etc.

**NES Memory Map:**
- `$0000-$07FF`: Work RAM
- `$2000-$3FFF`: PPU registers
- `$4000-$4017`: APU/I/O
- `$4018-$7FFF`: PRG ROM (switchable, mapper-dependent)
- `$8000-$FFFF`: **PRG ROM (fixed or switchable)** <- THIS IS WHERE THE CODE IS!

### Critical Bug in analyzer.cpp (Line 216)
```cpp
// Only analyze ROM space
if (offset >= 0x8000) continue;  // WRONG! This skips the ROM region for NES!
```

This check skips addresses >= 0x8000, which is exactly where NES PRG ROM is mapped!

### Interrupt Vectors Not Followed
For a.nes (Mapper 1, 128KB PRG):
- NMI vector ($FFFA): 0x0104 (in RAM - needs special handling)
- Reset vector ($FFFC): 0xC000 (in ROM at $8000-$FFFF - SKIPPED by bug!)
- IRQ vector ($FFFE): 0x0314 (in RAM - needs special handling)

The reset handler at 0xC000 is never analyzed because the analyzer skips all addresses >= 0x8000.

### ROM::read_banked() Also Uses Wrong Memory Map
The function assumes ROM is at `$0000-$7FFF` (GameBoy style):
```cpp
uint8_t ROM::read_banked(uint8_t bank, uint16_t addr) const {
    if (bank == 0 || addr < 0x4000) {
        // Bank 0 or fixed bank area - reads from start of file
        ...
    } else {
        // Switchable bank area (0x4000-0x7FFF) - uses bank offset
        ...
    }
}
```

For NES, ROM is at `$8000-$FFFF`, so this function reads from wrong file offsets.

### Evidence from Debug Run
```
Without aggressive scan:
  Found 7 functions (all at 0x4000 in banks 1-7)
  Found 13 basic blocks
  95 instructions

With aggressive scan:
  Found 13063 false "entry points" in RAM regions
  Drowns out real entry points
```

### Required Fixes
1. **analyzer.cpp**: Change `if (offset >= 0x8000) continue;` to `if (offset < 0x8000) continue;` or remove it
2. **analyzer.cpp**: Properly handle NES memory map for bank switching
3. **rom.cpp**: Fix `read_banked()` to use NES memory map ($8000-$FFFF for ROM)
4. **analyzer.cpp**: Add proper handling for interrupt vectors in last ROM bank
5. **analyzer.cpp**: Disable or fix aggressive scan to avoid false positives

## PPU Debugging Findings (2026-03-12)

### Issue: SDL Window Shows Only Gray

**Root Causes Found and Fixed:**

1. **CHR ROM Data Not Loaded** (FIXED)
   - `nes_context_load_rom()` was not copying CHR ROM data from the ROM file to CHR RAM
   - The iNES file has: 16-byte header + PRG ROM + CHR ROM
   - CHR ROM starts at offset: 16 + (prg_rom_units × 16KB)
   - Fix: Parse iNES header, extract CHR ROM data, copy to CHR RAM

   **File Modified:** `/common/active/sblo/Dev/nes-recompiled/runtime/src/nesrt.c`
   ```c
   /* Copy CHR ROM data to CHR RAM */
   if (chr_rom_data && chr_rom_size > 0) {
       size_t copy_size = (chr_rom_size < chr_ram_size) ? chr_rom_size : chr_ram_size;
       memcpy(chr_ram, chr_rom_data, copy_size);
       DBG_PPU("Loaded %zu bytes of CHR ROM data", copy_size);
   }
   ```

2. **Reset Vector Not Read** (FIXED)
   - PC was initialized to 0x0000 and never updated from reset vector
   - 6502 CPU should read reset vector from $FFFC-$FFFD at power-on
   - Fix: Read reset vector from end of PRG ROM data in `nes_context_load_rom()`

   **File Modified:** `/common/active/sblo/Dev/nes-recompiled/runtime/src/nesrt.c`
   ```c
   /* Read reset vector from ROM ($FFFC-$FFFD) and set PC */
   size_t reset_file_offset = 16 + prg_rom_size - 4;
   uint16_t reset_vector = data[reset_file_offset] | (data[reset_file_offset + 1] << 8);
   ctx->pc = reset_vector;
   ```

3. **PPU Rendering Disabled by Game Code** (PARTIAL FIX)
   - Test ROM (a.nes) writes 0x00 to PPU registers $2000 (CTRL) and $2001 (MASK)
   - This disables both background and sprite rendering
   - Added temporary debug override to force-enable rendering if CHR data exists

   **File Modified:** `/common/active/sblo/Dev/nes-recompiled/runtime/src/ppu.c`
   ```c
   /* TEMPORARY DEBUG: Force-enable rendering if both are disabled */
   if (!(ppu->mask & (PPUMASK_SHOW_BG | PPUMASK_SHOW_SPR))) {
       if (ppu->vram[0] != 0 || ppu->vram[1] != 0) {
           debug_mask = PPUMASK_SHOW_BG | PPUMASK_SHOW_BG_LEFT;
           DBG_PPU("DEBUG: Forcing background rendering enabled");
       }
   }
   ```

4. **Recompiler Analyzer Skips ROM Region** (NOT FIXED - KNOWN ISSUE)
   - `analyzer.cpp` line 216: `if (offset >= 0x8000) continue;`
   - This skips the entire NES PRG ROM region ($8000-$FFFF)!
   - Only 7 functions found in 128KB ROM (should be hundreds)
   - Most execution falls through to interpreter
   - This is the MAIN blocker for proper ROM execution
